Still calculating pay details incorrectly

This is a continuation of ticket 0073, where the customer is stating that the suggested changes are not working.

Introduction

The below image represents the instructions from the client to helpdesk.
It appears that helpdesk has not understood the "Site Rate.." request.

Taking any of the 4 sites noted in the above image, we confirm that the "Permanent Auto Classification" is set at the same value as the "Casual Auto Classification", which is perhaps the issue leading to this continutation of incorrect pay rates.

In reviewing the requested values for the "Site Rate", we have concluded that there's a need to create 4 distinct and new classification codes to cover the requirements expressed above.

The new classifications have been given the names reflecting the sites to which they correspond, to make the linkages easy to follow.
An example of how to correct the configurations is shown below for site (543).

Ensuring that we double-check everything, we've checked the 2 employees in this issue, and confirm their employment types as shown below.

  • The customer note is clear that these are both intended to be full-time employees, yet you can see in the "PPT-SALARY" employment definition that the FTE value is 0.8.
    A full-time employee must have a value of 1.0 or greater for the award logic to use the "Permanent Auto Classification" grid (as per the image above), and be considered full-time.
    Until that change is made, this employee will continue to be viewed as "Casual" for the purposes of this exercise.

 

  • One the one employee that is correctly an FTE of 1.0, their original shift costs are shown immediately below.
    This is on the scheduling workbench - showing the employee's regular rate of pay.
    If you double-check anywhere within this Costs tab (to recost the shift), the system will re-work its Award Interpreter (AI) logic to pick up the now correct new rate of $26.10 (as shown in the lower part of the image below)
